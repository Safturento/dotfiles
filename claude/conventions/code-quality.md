# Code quality conventions

Applies to any project. Read once at the start of non-trivial work, then internalize.

## Cleanliness philosophy

Two gates, both mandatory:

- **Before every commit**, run the project's lint and format-check (e.g. `npm run lint && npm run format:check`). The working tree must be free of lint errors and formatter diffs *at every commit boundary* — not just at hand-off. Mid-work commits that defer cleanup accumulate cruft that's hard to bisect later and trains the muscle of "I'll fix it at the end" (which then doesn't happen).
- **Before reporting any task as complete (or opening a PR)**, additionally run typecheck and the test suite, and resolve everything they surface. This is the full cleanliness sweep — lint + format + typecheck + tests, all green.

Both gates apply to interactive work, not just autonomous dispatches. "I'll lint it before the PR" is not a substitute for cleaning each commit — by the time you reach the PR, you've lost the per-commit context for what introduced what.

- **Treat warnings as errors.** A clean baseline is the only useful baseline; once warnings are tolerated they accumulate and stop being read.
- **Fix the underlying issue rather than silencing it.** Don't reach for `eslint-disable`, `// @ts-ignore` / `@ts-expect-error`, `any`, or non-null assertions to make a rule go quiet. If a lint rule is firing, it's usually pointing at a real problem (unused code, missing dep in a hook, unsafe cast) — address that.
- **If you genuinely don't know how to fix a warning/error, stop and ask.** A one-line disable is a last resort that needs explicit sign-off and a comment explaining why.
- **Type errors get the same treatment** — fix the types, don't paper over them.

We don't use pre-commit hooks; agents are responsible for running lint / format / typecheck themselves at the gates above. There is no safety net to catch you.

## Git hygiene baseline

### `.gitattributes` — line endings

Every repo gets a `.gitattributes` that forces LF line endings (`* text=auto eol=lf` plus an explicit `*.sh` rule are the load-bearing lines). The full template, the diagnosis of why it matters on WSL/Windows, and a fixer script for worktrees that already got CRLF'd live in [`line-endings.md`](./line-endings.md) — read that when setting up a new repo or when troubleshooting CRLF symptoms.

### `.gitignore` — universal baseline

Patterns that belong in nearly every repo, regardless of language. Language-specific additions go in the relevant convention file (e.g. `node.md` for `node_modules` / `*.tsbuildinfo`).

```gitignore
# IDE / editor
/.idea
.vscode/*.local
*.swp
*.swo
*~

# OS metadata
.DS_Store
Thumbs.db
desktop.ini

# Local env / overrides / secrets
*.local
.env
.env.*

# Logs
*.log
logs

# Test outputs
coverage
.nyc_output
```

Per-project additions (e.g. `.crew/`, `*.sqlite`, `Claude-Design-Handoffs/private/`) belong in that project's `.gitignore`, not in this baseline.

### Refuse to clobber files we didn't generate

When a script writes to a file that might also be hand-edited (`.env`, `settings.json`, generated TOML, etc.), tag the generated version with a header — e.g. `# generated by crew` — and have the script refuse to overwrite a file missing that header. Trades a little ceremony for protection against silently destroying a hand-written config.
